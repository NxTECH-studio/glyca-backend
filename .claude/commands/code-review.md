# 変更コードの自動レビュー（並列エージェント版）

## 現在の状況
- **現在のブランチ**: `{{bash: git rev-parse --abbrev-ref HEAD}}`
- **Base ブランチ**: `main`
- **変更ファイル**:
```
{{bash: git diff main --name-only | head -30}}
```

## 実行手順

### Step 1: 事前準備
1. `.claude/review-patterns.md` を読み込み、過去のレビューパターンを把握
2. 変更ファイルの一覧を確認
3. 以下の `.rubocop.yml` 無効化ルール一覧を確認し、**エージェント 2, 4 のプロンプト内の `[RUBOCOP_DISABLED_RULES]` をこの一覧で置き換えてから**エージェントを起動すること

#### プロジェクトの .rubocop.yml 無効化ルール（Enabled: false）
```
{{bash: grep -B1 'Enabled: false' .rubocop.yml | grep -v 'Enabled: false' | grep -v '^--$' | sed 's/://g' | sed 's/^# .*//' | grep -v '^$' | sed 's/^/- /'}}
```

### Step 2: 並列レビューエージェントの起動
以下の **4つのエージェントを並列で起動** してください。各エージェントは専門の観点でレビューを行います。

**必ず Agent ツールを使って4つのエージェントを同時に起動すること。**

---

#### エージェント 1: セキュリティ・規約チェック
```
あなたはセキュリティと規約の専門家です。

以下のファイルの変更（git diff main...HEAD）をレビューしてください。

## チェック項目

### セキュリティ（OWASP Top 10 準拠）
- SQLインジェクション（生SQL、#{}埋め込み、unsanitized な order/pluck）
- 機密情報ハードコード（API key、パスワード、トークン）
- 認証漏れ（before_action での認証チェック）
- 認可漏れ（適切なアクセス制御が行われているか）
- Strong Parameters 未設定・過剰許可
- Mass Assignment（permit に不要な属性が含まれていないか）
- ログへの機密情報出力（filter_parameters 対象外の個人情報）

### 外部サービス連携（該当ファイルがある場合のみ）
- Webhook の署名検証が行われているか
- 外部APIレスポンスの適切なバリデーション

### 規約
- Rails migration 使用禁止（Ridgepole 必須。スキーマ変更は db/Schemafile に記述）
- API専用アプリのため、ビュー・アセット関連のファイルが含まれていないか

## 出力形式
問題があれば以下の形式で報告：
[ファイル名:行番号] 問題の概要
問題: 具体的な問題
影響: セキュリティへの影響
修正案: コード例

問題がなければ「セキュリティ・規約面で問題なし」と報告。
```

---

#### エージェント 2: パフォーマンス・設計・品質チェック
```
あなたはパフォーマンス、設計、コード品質の専門家です。

以下のファイルの変更（git diff main...HEAD）をレビューしてください。

## 重要: プロジェクトのRubocop設定（無効化ルール）
以下のルールはプロジェクトで意図的に無効化されているため、**これらに関する指摘はしないこと**：
[RUBOCOP_DISABLED_RULES]

特に注意：
- Metrics/MethodLength が無効化 → 「メソッドが長すぎる」という指摘はしない
- Style/RedundantSelf が無効化 → `self.` の除去を提案しない
- Style/GuardClause が無効化 → ガード節への書き換えを提案しない

## チェック項目

### パフォーマンス
- N+1クエリ（includes/preload 未使用）
- includesではなくpreload/eager_loadを使うべき箇所がないか
- 不要なクエリ発行（ループ内でのDB問い合わせ等）
- バックグラウンドジョブにすべき重い処理（Solid Queue を使用）

### 設計
- Service Object の責務過多
- Model の肥大化
- 依存関係の方向（上位層が下位層に依存しているか）
- 既存パターンとの一貫性（同様の機能が他にある場合、同じ設計パターンに従っているか）

### コード品質
- 重複ロジックがないか（メソッド抽出・変数化）
- 共通化できる箇所がないか
- エラーハンドリングの適切さ（rescue が広すぎないか、握りつぶしていないか）
- 戻り値の一貫性（同メソッド内で型が変わらないか）
- 不要な複雑さ（過剰なメタプログラミング、不要な抽象化）

## 出力形式
問題があれば以下の形式で報告：
[ファイル名:行番号] 問題の概要
問題: 改善が推奨される点
理由: なぜ修正すべきか
修正案: コード例

問題がなければ「パフォーマンス・設計・品質面で問題なし」と報告。
```

#### エージェント 3: テスト・カバレッジチェック
```
あなたはテストの専門家です。

以下のファイルの変更（git diff main...HEAD）をレビューしてください。

## プロジェクトのテスト環境
- RSpec + FactoryBot（create, build 等はメソッドとして直接利用可能）
- database_rewinder でDBクリーンアップ
- bullet でN+1検出
- let! を優先（RSpec/PreferLetBang が有効）

## チェック項目
- 新規機能に対応するspecファイルがあるか
- 正常系だけでなく異常系・エッジケースのテストがあるか
- 境界値テストがあるか
- モックの適切な使用
- テストが必要なケースをカバーしているか
- FactoryBot の定義が適切か

## 出力形式
問題があれば以下の形式で報告：
[ファイル名] 問題の概要
問題: テスト不足の内容
理由: なぜテストが必要か
推奨: 追加すべきテストケース

問題がなければ「テストカバレッジは適切」と報告。
```

---

#### エージェント 4: 命名・可読性 + Linter チェック
```
あなたは可読性・命名・Linterの専門家です。

以下のファイルの変更（git diff main...HEAD）をレビューしてください。

## 重要: プロジェクトのRubocop設定（無効化ルール）
以下のルールはプロジェクトで意図的に無効化されているため、**これらに関する指摘はしないこと**：
[RUBOCOP_DISABLED_RULES]

特に注意：
- Style/GuardClause が無効化 → ガード節への書き換えを提案しない
- Style/NegatedIf が無効化 → `unless` への書き換えを提案しない
- Style/RedundantSelf が無効化 → `self.` の除去を提案しない

## 命名・可読性チェック項目
- 引数名が意図を表しているか
- 条件判定が直接的か（間接的な判定を避ける）
- マジックナンバーをenum化しているか
- 可読性が保たれているか（複雑すぎるロジックがないか）

## Linter チェック
変更された .rb ファイルに対して RuboCop を実行してください：

git diff main --name-only --diff-filter=AM | grep '\.rb$' | xargs bundle exec rubocop

## 出力形式
### 命名・可読性
改善提案があれば以下の形式で報告：
[ファイル名:行番号] 提案内容
現状: 現在のコード
提案: 改善案

問題がなければ「命名・可読性は良好」と報告。

### Linter
違反があれば以下の形式で報告：
[ファイル名:行番号] ルール名: エラー内容
自動修正コマンド: bundle exec rubocop -A ファイル名

違反がなければ「Linter違反なし」と報告。
```

---

### Step 3: 結果の統合
各エージェントの結果を受け取り、以下の形式で統合レポートを作成：

```markdown
# コードレビュー結果

## 変更ファイル一覧
- [ファイル一覧]

## 🔴 Critical（要修正）
[セキュリティ・規約チェック、Linterチェックの結果から]

## 🟡 Warning（推奨修正）
[パフォーマンス・設計チェック、テストチェックの結果から]

## 💡 Info（改善提案）
[命名・可読性チェックの結果から]

## 🟢 Good
[良い実装の例]
```

### Step 4: すべての問題を修正（Critical + Warning + Info）

**重要: Criticalだけでなく、Warning・Infoレベルの問題もすべて修正する**

1. レポートの 🔴 Critical → 🟡 Warning → 💡 Info の順に**すべて修正**する
2. 明示的に「Criticalのみ修正」と指示されない限り、低重要度の問題もスキップしない
3. 修正完了後、以下のサマリーを出力する：

```markdown
# 修正サマリー

## 🔴 Critical: X件修正
- [修正内容]

## 🟡 Warning: X件修正
- [修正内容]

## 💡 Info: X件修正
- [修正内容]

## 修正なし（対応不要と判断）
- [理由とともに記載。基本的にゼロであるべき]
```

4. すべての修正をコミットする（未コミットファイルを残さない）
